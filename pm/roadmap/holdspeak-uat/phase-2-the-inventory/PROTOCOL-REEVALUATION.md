# UAT Protocol Re-evaluation (flotilla, 2026-07-09)

> Generated by an 8-agent audit (7 domains + protocol-conformance → synthesis)
> re-evaluating whether the packs test the right things against the shipped surface.
> The verdict drives the next round of authorship. See §6 for the ordered punch-list.

## 1. Verdict

**We are testing the honest-failure spine well and the intelligence spine barely at all.** Across all four packs the trust/degrade discipline is real — every pack closes on a beat that can fail loudly (Pack D is all-failure, `a-04` closes on unapproved-never-egresses, `c-04` on endpoint-dies-mid-pack, smoke has dedicated failure scenarios), and non-deterministic intel is correctly judged structurally rather than on wording. But the rule the protocol declares MANDATORY for every LLM feature — control-vs-treatment — is honored in exactly **one beat out of the entire corpus** (`c-01` grounded-rewrite), and even that crams control and treatment into a single YAML step. The result: the product's flagship *intelligence* claims (the aftercare digest, grounded ask-on-pile, multi-intent routing, the 14 typed artifacts, the learned-from-N chip) are exercised happy-path or not at all, so a silently-degraded LLM would ship green. Coverage is also thin against the directory (meetings 5/27 keys, mesh 3/41, desk can only stage 2 of 7 primitive types), and the three-surface parity claim — the rig's whole reason for existing — is *asserted, never forced*: the 212 iPhone unknowns are being closed by `n/a: "unverified in the record"`, which converts the exact question UAT exists to answer into a permanent opt-out. So: right *discipline*, wrong *coverage*. We are not yet testing the right things.

## 2. The biggest misses

Ranked by (must-test priority × how load-bearing × how badly current packs miss it).

1. **No control-vs-treatment on the aftercare digest** (`meetings.aftercare.digest`, MUST, LLM). Walked in `a-02` and `smoke-03` as pure happy-path. A digest that collapsed to boilerplate, ignored the KB, or hallucinated an owner passes green. *Close it:* generate the digest with the prior-meeting delta context DISABLED (control) then ENABLED (treatment); assert the treatment names the specific carried-over open action/owner the control cannot.

2. **The desk's flagship grounded-ask is claimed but never invoked** (`desk.ask-ai.grounded-on-pile`, MUST, LLM). `smoke-01` cites this key while its three steps only *open and read* notes — no lasso, no `/api/ask`, no unknown-id refusal. Coverage math is being lied to. *Close it:* lasso the seeded pile, Ask AI a pile-only question with grounding OFF then ON; treatment cites BLUEBIRD/Pylon the control can't; then reference an unknown id and assert refusal-by-name.

3. **The mesh handoff arc is never driven** (`mesh.handoff.web_run_to_node`, MUST). The domain's headline claim — author-on-iPad → run-on-hub → read-on-web — is asserted nowhere; `smoke-06`+`pack-d/06` only walk serve-then-die, duplicated. *Close it:* on `mesh-node-alive`, run a desk ask against the `uat-worker` profile; assert it returns badged `⇄ mesh` within deadline AND the worker log shows the claim while the hub log shows no model load.

4. **Preview-before-type is completely untested** (`dictation.pipeline.preview-before-type` / `dictation.preview_before_type`, MUST). This is the load-bearing "nothing types until you confirm" trust promise and NO scenario asserts it — not the verbatim fidelity, not the one-shot token burn, not the off-path byte-lock. *Close it:* enable preview, dictate; assert an armed card and that NOTHING types while armed; "Type it" lands exactly the previewed text verbatim once; Discard burns the token.

5. **Every trust-boundary claim is written in the imagined voice, never attacked** (`trust.binding.loopback-token-gate`, `trust.secrets.profile-key-never-syncs`, `trust.no-telemetry`, `release.schema.safe_upgrade` — all MUST). `pack-d/05` literally says the gate "would require one off-loopback" — a human imagines the hostile case; no recipe stages it, no probe fires it. A stranger never sees the gate reject anything. *Close it:* stage an off-loopback bind with no token → assert bind REFUSED; with a token, an off-loopback API route without it returns 401 while `/health` returns 200; push a profile key via REST body → re-read → assert the key field is absent in every response and ChangeSet; boot idle under network capture → assert zero beacons.

6. **The intel generation core is untested** (`meetings.intel.multi_intent_routing`, `plugins.14-typed-artifacts`, `meetings.intel.cloud_endpoint` — all MUST). The structural verdict stops at "action list + decisions"; it never asserts the substance the directory names (confidence scores, resolvable transcript sources, a concrete typed artifact like ADR/mermaid/`incident_timeline`/`risk_heatmap`), and the single seed `pylon-incident-meeting.vtt` is single-intent so MIR can't even be demonstrated. *Close it:* import a mixed-topic transcript; assert ≥2 plugin chains ran with per-segment lineage, a buried incident intent still routed, and each card shows a confidence score + a transcript-source ref that resolves.

7. **The voice-command macro side-effect seam is untested** (`macros.voice_commands.board` / `.fire_while_dictating`, MUST — the first production side-effect). Configuring-as-consent, shell-danger, whole-utterance-only matching, and "fires the action, types NOTHING" are all unasserted. *Close it:* add an echo shell macro on `/commands`, Test it fires; speak the keyword mid-dictation → action runs, zero text typed; off/no-match dictation is byte-identical.

8. **The learned-from-N chip has no honest-count assertion** (`dictation.learning-digest.honest-count`, MUST — "honest-by-construction" is a named pillar). `c-03` says "not an inflated one" but never asserts the zero case or a known-N; a digest hardcoded to "learned from 2" passes. *Close it:* open the digest at N=0 → assert honest zeros; make a correction matching exactly 2 past dictations → assert the chip reads "learned from 2 similar".

9. **The cross-machine steering + destructive-kill consent negatives are unstageable** (`steering.cross_machine`, `factory.kill`-refuses-unarmed, `mobile.mesh.serve` forced-run 0.00s refusal — MUST). The "refuse fast, by name" crown cases a stranger's hands must not falsify are absent, and the recipes to stage them don't exist. *Close it (once recipes exist):* relay arm+C-c+steer to a configured node → far node writes its own audit row, local does not; point at an offline node → `node_offline` by name in <5s; attempt Kill while UNARMED → refusal, session survives.

10. **The egress badge cloud-flip is never walked** (`trust.egress_badge`, MUST). Every badge scenario opens a local note and confirms "local"; the falsifiable FLIP to cloud-with-a-named-target is exercised on no surface. Also a directory-vs-product mismatch: the directory promises a *per-card* pill but web `egressBadge()` is one global chrome pill. *Close it:* create a GitHub-issue proposal card → assert its pill names the target with no prose; if per-card egress isn't implemented on web, record the mismatch in the ledger rather than asserting a per-card truth the code doesn't provide.

## 3. Mis-targeted tests

Scenarios that test the wrong thing, with the fix.

- **`smoke/01-desk-seeded-walk.yaml` — capability inflation.** Tags `desk.ask-ai.grounded-on-pile` (MUST, LLM) but every step only opens/reads; Ask-AI is never invoked and the pack is all-green (violates the no-all-green rule). *Fix:* add a real lasso→Ask-AI beat with an OFF/ON control and an unknown-id refusal close, or drop the tag so coverage math stops falsely crediting it.

- **`pack-c/01-grounded-rewrite.yaml` — right shape, wrong granularity.** The one true control-vs-treatment beat crams both legs into a single do/expect step ("dictate the same instruction twice — once OFF, once ON"), yielding one verdict for two runs so a per-surface record can't show which leg failed. *Fix:* split into two ordered verdicted beats — beat 1 (control, OFF): "no context"; beat 2 (treatment, ON): the repo token surfaces.

- **`pack-c/03-correction-ritual.yaml` — happy-path LLM feature, no control, no concrete N.** Asserts "the next dictation reflects it" and "reads the real count" with no OFF-vs-ON contrast and no pinned N to check against. *Fix:* dictate the similar utterance BEFORE the correction (assert wrong routing), correct, dictate again (assert right routing) — before/after IS the control; pin a known count; add an OFF leg asserting byte-identical routing when `session_memory` is disabled.

- **`pack-c/02-spoken-symbols.yaml` — deterministic feature that can't fail.** Asserts "one combined matcher pass, no double substitution" but never stages the double-substitution trap it names, so the very regression the expect line calls out cannot fail the beat. *Fix:* add an adversarial phrase engineered to double-substitute under a two-pass matcher (the Phase-59 gotcha) and assert it lands once.

- **`pack-a/01-import-to-artifacts.yaml` — structural verdict too shallow, no control.** "Right artifact types rendered" never asserts confidence, resolvable source, or a concrete typed artifact, and nothing proves the artifacts came from the live LLM vs seed/fallback. *Fix:* assert confidence + resolvable-source per card, name at least one concrete typed artifact (`incident_timeline` for the pylon seed), and pair with an endpoint-down control that produces an honest empty.

- **`pack-a/02-digest-and-moment.yaml` — the delta claim exercises nothing.** `meeting-just-ended-open-actions` imports ONE meeting, so the "delta since the previous meeting" path (aftercare.py:42-44) is vacuous at N=1. *Fix:* stage a two-meeting world so the since-last delta is assertable; add the empty-quiet control (a meeting with nothing open stays quiet, no padding).

- **`pack-d/04-egress-badge-local.yaml` + `smoke/07` — happy-path in the local direction only, and per-card assertion the code doesn't back.** Both open a note and assert "local"; the load-bearing cloud flip is never walked, and treating the badge as per-card asserts nothing the note controls when web `egressBadge()` is global chrome. *Fix:* add the cloud-direction beat, or record the badge-is-chrome-global mismatch.

- **`pack-d/05-key-never-syncs.yaml` — conditional voice + prose verdict.** Step 2 asserts the loopback gate as "would require one off-loopback" (imagined, unstaged); step 1 asserts the privacy chip's prose is "without marketing" rather than tying it to real config. *Fix:* stage the off-loopback adversarial request backed by a probe; make the chip control-vs-treatment on config (read local-only, then configure a cloud endpoint → chip changes and names the endpoint).

- **`smoke/06` + `pack-d/06` — near-duplicate, no treatment beat.** Same recipe, same serve-then-die beats, same 3 features — the domain proves the node lives and dies but never that a run moves onto it and returns badged; and `pack-d/06`'s "refuse fast, by name" is eyeballed while the intel dead-endpoint probe machine-checks `elapsed<5s`. *Fix:* repoint `smoke/06` at the handoff (badged return + worker-log provenance), keep `pack-d/06` as the offline close, and add a <5s timing probe to the forced mesh run.

- **`pack-c` — the whole device column silently punted.** All four scenarios `n/a` iPad AND iPhone with "dictation depth is unverified in the record," but the directory rows for `dictation.pipeline.intent-routing` and `dictation.pipeline.dry_run` say **ipad: yes** with iPad-specific hints, and a whole `mobile.dictation.*` family is **iphone: yes**. This is a *false n/a* that contradicts the source of truth. *Fix:* flip the iPad leg to yes for intent-routing/dry_run/action-to-issue and add the "preview matches what types verbatim" iPad beat; keep iPhone n/a ONLY where the inventory row is genuinely unknown.

## 5. The three-surface reckoning

**The parity claim is being asserted, not tested — and the rig knows it.** The machinery is right: `uat/conductor/contract/coverage.py` correctly refuses to credit a surface a pack never walks, `scenarios.py` requires a stated reason for every `n/a` and refuses a beat that is `n/a` on all three surfaces, and the ledger honestly carries the 212 iPhone-unknown / 97 iPad-unknown cells. The problem is authorial, not structural: only **4 of ~22 scenarios** set `iphone: yes`, and they all cluster on the egress badge / privacy chip — exactly **1 of the 25 inventory-confirmed iPhone-yes capabilities** (`mobile.trust.egress.badge`) is touched by any beat. Every other iPhone `n/a` reason is a variant of "unverified in the record," which is the definition of an **unknown**, not an `n/a`. `n/a` should mean "this surface genuinely lacks the capability" (cite the `no`/`absent` row); "we haven't looked yet" must become a *forced beat*, not an opt-out. Worst case: `pack-c` — the pack PHASE-3-PLAN names as "where the device column gets answered" — punts BOTH iPad and iPhone under a false `n/a` that directly contradicts `ipad: yes` rows in the source of truth.

**What it would take to answer the columns:** (1) build the device pre-flight block (`model-installed-on-device`, `hub-reachable-on-lan`, `two-devices-paired`) and wire it into `pack-c` and a new Pack E so the phone dictation + phone mesh-serve legs are *forced*; (2) flip `pack-c`'s false iPad `n/a` to yes for the rows the record supports and add the "preview matches what types verbatim" iPad beat; (3) add a lint to `build_ledger`/`scenarios.py` that **refuses an `n/a` reason of the form "unverified/unknown" for a capability whose inventory row is `iphone: yes`** (ERROR: false n/a); (4) add a per-pack coverage assertion that fails a pack whose iPhone coverage over `iphone: yes` capabilities falls below threshold, making "phone column answered" a machine-checked exit condition. Until then the sitting ships "whole" while the plan's highest-information column stays blank behind green `n/a` badges.
