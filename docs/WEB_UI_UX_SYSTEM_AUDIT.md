# HoldSpeak Web UI/UX system audit

Status: active design contract

Scope: the one Vite/React Web product

Out of scope: native Swift DeskOS acceptance

The Web uses this audit as its measured baseline. Every route now shares one
Vite/React application, Signal component grammar, API client, and runtime bus.

## The decision

HoldSpeak Web has one interaction language: **Signal**. Pages may have different
information architecture, but they may not privately redefine the basic shape,
size, type, focus, error, disabled, or loading behavior of buttons, editors,
checkboxes, selects, and status messages.

Compact browser screenshots are evidence for responsive Web behavior only. They
are not evidence for the separate Swift DeskOS implementation. Native DeskOS
must be built and tested on its actual Apple target.

## What the audit found

The pre-migration source inventory contained 140 inputs, 33 selects, 11
textareas, and 321 buttons across two rendering systems. The heaviest Web
surfaces were Settings, History, Activity, Live, Profiles, Welcome, and the
Desk inline editor.

The first automated browser pass captured 17 routes at 1440×1000 and 430×932.
It found:

- Page-local copies of button and field CSS in Settings, History, Activity,
  Live, Setup, Welcome, Dictation, dialogs, and the React Desk.
- JavaScript-rendered Activity actions falling back to raw browser buttons
  because runtime-created markup missed the page-local style contract.
- Raw or undersized checkboxes in Activity, History, and Settings.
- A native select and unlabeled project-root editor in Dictation.
- Inconsistent action heights and baseline alignment in discovery forms.
- Internal transport output such as `Found 1 model.` presented like product
  feedback even though the selected model already communicates success.
- Multiple style signatures for the same control family on a single route.
- The living component gallery was build-accessible but had no first-class
  product-server route, so the audit received a 404.

The screenshots and machine-readable measurements are generated by
`scripts/web_ui_audit.py`. The report deliberately calls its narrow viewport
`compact_web`, never iPad or iOS.

## Product principles

1. **One component, one contract.** Shared controls own dimensions and states;
   pages own placement and composition.
2. **Visible state, not debug narration.** Status text exists when it confirms a
   completed user task, explains a blocking condition, or gives a recovery
   action. Counts already visible in the UI are not repeated as prose.
3. **Labels survive placeholders.** Every editor has an associated visible
   label, with concise hint and error text when needed.
4. **The row aligns to the control.** An action beside a labeled field aligns
   with the editor, not the label or the entire field stack. Primary actions use
   the same 44 px control height.
5. **Choice mechanics match choice count.** Use direct choices/cards for a
   handful of important options, a select for a moderate familiar set, and a
   searchable combobox for a large or remote set.
6. **Native semantics, designed presentation.** Preserve real inputs, buttons,
   labels, fieldsets, and disclosure semantics. Styling must not replace the
   keyboard and assistive-technology contract.
7. **Local-first trust is explicit.** Explain where a credential goes and what
   can leave the machine near the relevant control, without exposing secrets.
8. **Web and Swift share product intent, not implementation evidence.** Their
   test matrices and platform controls remain distinct.

## Canonical component grammar

| Family | Component/class | Size | Required behavior |
|---|---|---:|---|
| Text/number/url | `FormField` + `.hs-control` | 44 px | visible label, hover, focus ring, disabled, hint/error association |
| Moderate select | `.hs-control.hs-select` | 44 px | native semantics, Signal chevron, no platform chrome |
| Important small choice set | choice cards/radios | ≥44 px target | full row clickable, selected mark plus color |
| Checkbox | `Checkbox` / `.hs-check` | 20 px mark; ≥44 px row target where practical | full label clickable, focus visible, checked mark independent of color |
| Primary/secondary button | `Button` / `.btn` | 44 px | shared type, radius, disabled/loading/focus/press states |
| Dense subordinate button | `.btn--sm` | 36 px | only inside toolbars or repeated rows; never a lone primary form action |
| Disclosure | native `details/summary` or APG disclosure button | ≥44 px target | `aria-expanded` when button-based, keyboard operable |
| Status | `InlineMessage` or scoped live region | content-driven | success, warning, error, or progress; no transport/debug echo |

Spacing is limited to the existing 4/8/12/16/24/32/48/64 px token scale.
Fields use 8 px internal label/control/hint rhythm. Related fields use 16 px.
Sections use 24 to 32 px. A control and its adjacent action use 12 px.

## Status-message policy

Show status when the user needs new information:

- `Connected to Office server.` after an explicit connection test.
- `No models were reported. Enter a model manually.` with a recovery action.
- `Could not reach the server.` plus the actionable reason and retry affordance.
- `Saving…` and `Saved` when persistence is not otherwise obvious.

Do not show:

- `Found 1 model.` directly beneath the model that is already selected.
- Raw HTTP status, response shape, internal route, or polling output.
- Success copy that shifts layout without changing what the user can do.

## Evidence and standards

- WCAG 2.2 AA requires pointer targets to be at least 24×24 CSS px or satisfy
  its spacing exceptions; the enhanced target is 44×44. HoldSpeak uses 44 px
  for primary editors/actions and 36 px only for dense subordinate actions:
  [Target Size (Minimum)](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html),
  [Target Size (Enhanced)](https://www.w3.org/WAI/WCAG22/Understanding/target-size-enhanced).
- WAI form guidance calls for visible labels, related-control grouping, clear
  instructions, validation, and notifications:
  [Forms tutorial](https://www.w3.org/WAI/tutorials/forms/),
  [Labeling controls](https://www.w3.org/WAI/tutorials/forms/labels/),
  [Validating input](https://www.w3.org/WAI/tutorials/forms/validation/),
  [User notifications](https://www.w3.org/WAI/tutorials/forms/notifications/).
- Checkbox, listbox, and disclosure behavior follows the WAI-ARIA Authoring
  Practices patterns:
  [Checkbox](https://www.w3.org/WAI/ARIA/apg/patterns/checkbox/),
  [Listbox](https://www.w3.org/WAI/ARIA/apg/patterns/listbox/),
  [Disclosure](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/).
- USWDS recommends select controls for a moderate option set, radio controls
  for fewer than about seven choices, and a combobox for more than about 15:
  [Select](https://designsystem.digital.gov/components/select/),
  [Checkbox](https://designsystem.digital.gov/components/checkbox/),
  [Form](https://designsystem.digital.gov/components/form/).
- Apple emphasizes hierarchy, harmony, consistency, and platform-aware
  controls; the Web uses those principles without pretending browser evidence
  validates Swift:
  [Design principles](https://developer.apple.com/design/human-interface-guidelines/design-principles),
  [Buttons](https://developer.apple.com/design/human-interface-guidelines/buttons).

## Migration order

### P0: visible breakage

- Remove discovery-count debug copy and align discovery actions.
- Establish global button, editor, checkbox, focus, and disabled baselines.
- Fix raw runtime-created Activity controls.
- Give the component gallery a first-class route.
- Eliminate unlabeled and sub-24 px controls.

### P1: editor-heavy surfaces

- Settings: shared field/checkbox/button components and section rhythm.
- History: shared facets, filters, dialogs, and repeated row actions.
- Activity: shared rules/connectors forms and rendered action toolbars.
- Dictation: shared project-root editors, tabs, and runtime configuration.
- Workbench: shared toolbars while preserving graph-specific canvas controls.

### P2: convergence

- Profiles, Live, Commands, Setup, Companion, and secondary surfaces.
- React Desk editor primitives aligned to the Signal tokens and state grammar.
- Remove obsolete page-local control definitions after each migration.

## Enforcement

Every Web change that adds or modifies a control must:

1. Render the component/state in `/design/components` when it introduces a new
   reusable pattern.
2. Pass production build and focused integration tests.
3. Run `scripts/web_ui_audit.py` against the live product server.
4. Introduce no raw native select, unnamed control, sub-24 px target, console
   error, or new per-route style signature without a documented exception.
5. Be verified separately on the native target if it changes Swift DeskOS.
