# Guru Meditation: HoldSpeak

Date: 2026-06-07

## Interrupt Code

HoldSpeak is not a toy anymore. It is a pre-release local-first voice operating
loop with several real subsystems:

- hold-to-talk dictation into any app
- project-aware rewriting and routing
- a persistent dictation journal with correction and replay
- meeting capture plus LLM-backed artifact synthesis
- a local activity/context layer
- opt-in actuators with approval gates
- an AIPI-Lite companion/device path
- a web-first runtime and Astro frontend

The project's center of gravity is strong. The danger is not lack of capability.
The danger is that the product has more capability than a new user can mentally
load, and the runtime still concentrates too many responsibilities in one place.

My read: this codebase is alive, ambitious, and better tested than most projects
at this stage. It now needs product semantics and runtime separation more than it
needs another feature.

## The Good

### 1. The product has a real soul

The strongest idea is not "voice typing." The strongest idea is:

> Say something rough. The local machine hears it, understands the work context,
> transforms it, remembers the attempt, lets you correct it, and can replay the
> same utterance through a better pipeline later.

That is a more durable idea than a dictation utility. It is a local memory loop
for spoken work.

The dictation journal, correction memory, and replay path are the project's most
important differentiator. They turn "AI made a mistake" into an inspectable,
teachable feedback cycle.

### 2. The privacy stance is unusually concrete

`docs/SECURITY.md` names data classes, trust boundaries, egress points, secrets,
and accepted residual risks. That is the right level of seriousness for a tool
that stores transcripts, activity, and voice-adjacent speaker embeddings.

The local-first posture is not just marketing. It is reflected in config defaults,
egress tests, loopback defaults, explicit connector permissions, device PSKs, and
human approval around external writes.

### 3. The architecture has been actively repaired

This repo has already survived several necessary amputations and decompositions:

- `db.py` became a `holdspeak/db/` package with domain repositories.
- `web_server.py` route responsibilities moved into route modules and `WebContext`.
- the TUI and menubar were removed rather than kept as zombie surfaces.
- audio ownership converged into `VoiceTypingSession`.
- meeting sessions became web-free event emitters.

That matters. It means the project is not merely accreting features. It is also
removing bad older assumptions.

### 4. The tests give you room to move

The test suite is broad enough to support serious simplification. There are unit,
integration, e2e, drift-guard, route, security, device, pipeline, and runtime
tests. The suite is not just checking pure functions; it exercises contracts and
user-visible flows.

That is a strategic asset. Use it to simplify aggressively where behavior is
covered.

## The Trouble

### 1. `WebRuntime` is the next central chip under thermal load

`holdspeak/web_runtime.py` is now the sole interactive runtime. It owns:

- startup and shutdown
- config application
- web server wiring
- hotkey callbacks
- device voice paths
- meeting lifecycle
- dictation transcription and typing
- desktop presence sync
- MIR controls
- plugin queue processing
- project association
- action updates
- activity broadcasting

Classifying the old function into `WebRuntime` was a good rescue. But the class
is still too broad. It is now the place where unrelated concerns meet, which
makes every new user-facing behavior tempted to become one more method.

Explicit advice: do not rewrite it. Split it by extracting small owned
controllers behind stable methods:

- `DictationRuntimeController`: hotkey/device voice stop -> transcribe ->
  process -> type -> journal/milestone.
- `MeetingRuntimeController`: start/stop meeting, bookmarks, title/tags,
  meeting save/result handling.
- `DeviceRuntimeController`: AIPI device events, health, queries, status
  pushback.
- `IntelRuntimeController`: MIR preview/history, deferred plugin queue,
  synthesis/artifacts.
- `PresenceRuntimeController`: runtime activity and desktop presence sync.

Keep `WebRuntime` as the composition root. The goal is not more abstraction. The
goal is to make the current responsibilities visible and testable in isolation.

### 2. The product vocabulary is ahead of the user

The active Phase 47 diagnosis is correct: "Project KB" and "Project Context" are
different mechanisms with confusingly adjacent names.

That issue is not cosmetic. It is a sign that the system has outgrown internal
phase vocabulary. A user should not need to know DIR-01, `kb-enricher`,
`project-rewriter`, `.holdspeak/project.yaml`, and `.hs/` before getting value.

Explicit advice: make Phase 47 the next real work. Do not skip it for a more
technical phase. If the "teach HoldSpeak this project" loop becomes legible, the
entire product becomes easier to sell, dogfood, and debug.

### 3. Some root planning docs are now traps

`TODO.md` still says "Make the TUI worth showing off." `HOLDSPEAK_REFACTORING.md`
still treats TUI cleanup as an active architecture track. But the TUI was killed
in Phase 32 and the changelog says the web runtime is the only interactive
runtime.

This creates a false signal for agents and contributors.

Explicit advice: move stale root planning docs into an archive or add a hard
banner at the top:

> Historical. Superseded by `pm/roadmap/holdspeak/README.md`.

Do this before more agent-driven work. Agents will read these files and believe
them unless the files tell the truth.

### 4. The frontend is becoming dense in the old web-app way

The Astro pages are rich and the design system has a coherent identity. But large
page files and large Alpine scripts are accumulating:

- `history.astro`: about 2,441 lines
- `dictation-app.js`: about 2,194 lines
- `dictation.astro`: about 2,065 lines
- `dashboard-app.js`: about 1,527 lines
- `index.astro`: about 1,368 lines

This is not yet a crisis. But if Phase 47 adds richer project-knowledge flows,
the dictation page will become the next overloaded surface.

Explicit advice: for Phase 47, do not just append more panels to
`dictation.astro`. Introduce smaller page-local components or section partials
where Astro permits it, and split `dictation-app.js` by behavior modules if it can
be done without fighting Alpine. No framework migration. Just keep the file from
becoming a second runtime monolith.

### 5. The database is greenfield-clean, not release-stable

The DB decomposition is good. The schema was intentionally squashed back to
`SCHEMA_VERSION = 1`, which is valid for a pre-release, greenfield posture.

But this is also a line in the sand: once HoldSpeak is actually released to users,
schema migration becomes a real product promise.

Explicit advice: before PyPI or a tagged public release, define the migration
policy. It can still be simple, but it must be explicit:

- what config/database versions are supported
- whether destructive migration is ever allowed
- how users back up or export before upgrade
- what `doctor` reports when schema state is unexpected

## Strategic Reading

HoldSpeak has three possible futures:

### Future A: The local dictation copilot

This is the highest-leverage path.

Make the everyday loop excellent:

1. Hold key.
2. Speak roughly.
3. It types the right thing into the right target.
4. If wrong, correct in the moment.
5. The system remembers locally.
6. Replay proves improvement.

This future should be the product's north star.

### Future B: The meeting intelligence workbench

This is already strong, especially with 14 real plugins, artifacts, proposals,
history, and action review. It is impressive, but it is more episodic than
dictation. Users may use meetings a few times a week. They use typing all day.

Keep investing, but do not let meeting intelligence pull focus from the daily
dictation loop.

### Future C: The ambient hardware companion

AIPI-Lite is charming and strategically interesting. But it is hardware-shaped,
network-shaped, and support-heavy.

Keep it as a companion path. Do not let it become the core adoption path. The
core should work beautifully on a laptop with a mic and no device.

## Correction To This Meditation

The first version of this meditation over-indexed on project hygiene. That was
too small a reading.

Yes, stale planning docs can mislead agents. But old PM artifacts are snapshots.
They are allowed to decay. They are not the product. Spending serious energy
polishing old planning history is a low-value move unless the file is actively
being read as canon and causing bad work.

The better question is:

> What can HoldSpeak do next that makes an end user feel, in one sitting, "I want
> this in my daily loop"?

The answer is not "update project management documentation." The answer is to
make the product more useful, more legible, more delightful, and more obviously
local-first at the exact moment the user is trying to get work done.

## Functional Bets Worth Making

### 1. The 10-minute daily-driver win

HoldSpeak should give a new user a useful result in ten minutes without reading
architecture docs or editing files.

Build a path like this:

1. Launch.
2. Pick local or OpenAI-compatible model.
3. Test mic.
4. Dictate into a real target.
5. See exactly what was heard, routed, rewritten, and typed.
6. Correct one mistake.
7. Repeat the same utterance and watch it improve.

That is not just onboarding. That is the product thesis in miniature.

### 2. Voice macros and command grammar

Project-aware rewriting is powerful, but users also need quick deterministic
wins. Add a small, visible "voice commands" layer:

- "new paragraph", "bullet list", "code block", "quote this"
- "send it", "copy only", "do not submit"
- "make it concise", "make it friendlier", "make it a GitHub comment"
- user-defined phrases like "standup update" or "bug report template"

The key is to make common spoken workflows feel immediate, not hidden behind LLM
magic. Deterministic commands should be inspectable and editable from the UI.

### 3. Correction should become a first-class ritual

The journal and replay path is the project's jewel. Push it harder:

- after every dictation, show a small "right / wrong" affordance in the cockpit
  and presence surface
- let the user correct target, wording, command mode, or template separately
- show "learned from 3 similar corrections" as a trust-building signal
- add a weekly "what HoldSpeak learned" view

Do not bury learning in a tab. Make the correction loop feel like training a
tool, not debugging a failure.

### 4. Project knowledge should feel like teaching, not configuration

Phase 47 is still the right next product work, but the framing matters. Do not
present "Project KB" and "Project Context" as two config editors. Present one
concept:

> Teach HoldSpeak how this project talks.

Then expose two plain sub-parts:

- **Facts**: names, commands, repos, URLs, acronyms, people, products.
- **Writing guidance**: how to phrase tasks, comments, shell commands, agent
  prompts, and issue updates for this project.

Keep the underlying `.holdspeak/project.yaml` and `.hs/` mechanics if needed.
The user should not have to care. The UI should ask useful questions and write
the files.

### 5. Meeting mode needs "aftercare," not more artifact types

The meeting side already has plugins and artifacts. The next user value is not
plugin count. It is follow-through:

- "What changed since last meeting?"
- "What did we decide?"
- "What is still open for me?"
- "Draft the follow-up message."
- "Turn the accepted actions into issues."
- "Show me the transcript moment that justifies this action."

Make meeting intelligence close loops. A beautiful artifact that never changes a
user's next action is decoration.

### 6. Local activity should become pre-briefing fuel

The browser/activity layer is powerful but abstract. Convert it into a concrete
user-facing win:

- before a meeting: "Here is what you touched since last time"
- before dictating into a repo: "I see this project; use its vocabulary?"
- after browsing a GitHub issue: "Want to dictate a reply with this issue as
  context?"

This is where HoldSpeak can feel ambient without being creepy. The rule: every
activity-derived suggestion must explain its source and stay dismissible.

### 7. Make privacy visible at decision points

The security posture is strong, but end users need to feel it.

Every place that could use a model, connector, actuator, device, or activity
source should answer three questions in plain language:

- What data is used?
- Does anything leave this machine?
- What control do I have right now?

That is a delight feature for this product category. It reduces fear and
increases trust.

### 8. Build "show me why" everywhere

For a local AI work tool, explainability is not academic. Add "why did it do
that?" affordances:

- why it picked a target profile
- why a block matched
- why a project fact was inserted
- why an LLM rewrite changed text
- why an artifact/action was proposed

This turns surprising behavior into inspectable behavior. Users forgive mistakes
when they can see the machine's reasoning path and correct it.

## Things Not Worth Worshipping

Do not spend meaningful product time on:

- making historical PM snapshots perfectly current
- preserving retired runtime concepts in docs
- adding more internal phase ceremony
- expanding plugin count for its own sake
- polishing architecture diagrams while the first-run value path remains unclear
- building hardware-first flows before the laptop-only flow is excellent

The only documentation work that matters right now is documentation that helps a
user succeed or prevents a contributor from doing the wrong work. Everything else
can be historical, imperfect, and clearly labeled as such.

## The Explicit Advice

Do this next:

1. **Ship Phase 47 before adding new capabilities.**
   But ship it as an end-user feature, not a docs cleanup. The user-facing
   promise is "teach HoldSpeak how this project talks." The output should be a
   guided setup and visible improvement in dictation quality.

2. **Make correction and replay impossible to miss.**
   Put the learning loop in the user's path. "That was wrong" should be a normal
   interaction, not a hidden maintenance flow. The payoff is: correct once,
   improve next time, prove it with replay.

3. **Add a small voice-command layer.**
   Give users deterministic, editable wins: formatting, submit/copy behavior,
   rewrite modes, and user-defined phrases. This will create daily utility even
   when model-powered rewriting is off or slow.

4. **Push meeting intelligence toward follow-through.**
   Less "here are more generated artifacts"; more "here is what to do next, here
   is the evidence, here is the approved action." Close loops.

5. **Extract from `WebRuntime`, one controller at a time.**
   Start with dictation because it is the daily-driver loop and Phase 47 will
   touch nearby surfaces. Keep all routes and callbacks stable. Let tests prove
   equivalence.

6. **Protect the local-first promise as a product feature, not just a policy.**
   The trust chip/status work is good. Keep making egress and storage state
   visible where decisions happen: model selection, connector enablement,
   actuators, project context, and device pairing.

7. **Only touch stale PM docs enough to stop active confusion.**
   Add a banner or move them aside if agents read them. Do not spend product
   energy making historical snapshots accurate.

8. **Prepare a release contract only after the above.**
   A public release before project-knowledge clarity and schema policy would
   convert current ambiguity into user support load.

- Draft the release-readiness contract only after the new user value path works.

## North Star

The best version of HoldSpeak is not "Whisper plus a web dashboard."

The best version is a local speech-to-work loop:

> It listens when asked, acts in the right context, explains what happened,
> remembers corrections, and never hides where the data went.

Build toward that. Cut or defer anything that does not make that loop clearer,
faster, safer, or more teachable.
